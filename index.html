react生命周期

发表于 2019-08-27 | 分类于 react
  • initialization初始化阶段:
    • constructor方法用来做一些组件的初始化工作
  • Mounting挂载阶段:
    • componentWillMount:在组件挂载到DOM前调用，且只会被调用一次，在这边调用this.setState不会引起组件重新渲染，也可以把写在这边的内容提前到constructor()中，所以项目中很少用。
    • render
      • 一个纯函数（PureFunction：函数的返回结果只依赖于它的参数；函数执行过程里面没有副作用）

      • 不能在里面执行this.setState，会有改变组件状态的副作用
    • componentDidMount:组件挂载到DOM后调用，且只会被调用一次

高阶函数与高阶组件

发表于 2019-07-04 | 分类于 react

高阶函数

满足以下条件之一就可称为高阶函数

  • 函数可以作为参数被传递
  • 函数可以作为返回值输出
  • 场景：
    • 防抖: 指的是无论某个动作被连续触发多少次，直到这个连续动作停止后，才会被当作一次来执行.比如一个输入框接受用户不断输入，输入结束后才开始搜索。每次操作时，清除上次的定时器定义新的定时器，一段时间后进行操作。
    • 节流: 指的是无论某个动作被连续触发多少次，在定义的一段时间之内，它仅能够触发一次。比如resize和scroll时间频繁触发的操作，如果都接受了处理，可能会影响性能，需要进行节流控制

性能优化之图片

发表于 2018-10-05 | 分类于 性能优化

针对不同的场景使用不同的图片格式

  • jpg有损压缩，压缩率高，不支持透明(应用于大部分图片不需要透明的业务场景)
  • png支持透明，浏览器兼容性好(大部分图片需要透明图片的业务场景)
    • png8 256(2^8)色 支持透明
    • png24 2^24色 不支持透明
    • png32 2^24色 支持透明
  • gif 使用于loading动画,只支持全透明和全不透明
  • webp压缩程度更好，在ios有兼容性问题(适用于安卓)
  • svg矢量图，代码内嵌，相对较小(图片样式相对简单场景),放大和缩小的时每一个像素会根据向量重新计算，固不会失真
  • css雪碧图(减少了 http 请求，有时候雪碧图太大，请求缓慢，还需要对雪碧图进行拆分，不适合大图合并)
  • base64(inline到html里，把图片内容内嵌到htm中)
    • 无额外请求，无需考虑跨域问题
    • 缺点：内嵌在html或css里代码可读性不好
  • iconfont(矢量图，字体文件取代图片文件，来展示图标、特殊字体等元素)
    • 减少 http 请求
    • 可以直接通过 css 的 font-size，color 修改它的大小和颜色，对于需要缩放多个尺寸的图标，是个很好的解决方案
    • 支持一些 css3 对文字的效果，例如：阴影、旋转、透明度
    • 缺点：操作流程繁琐

重绘与回流

发表于 2018-05-08 | 分类于 性能优化

回流

  • 当render tree中的一部分(或全部)因为元素的规模尺寸，布局，隐藏等改变而需要重新构建，称为回流
  • 当页面布局和几何属性改变时就需要回流

重绘

当render tree中的一些元素需要更新属性，而这些属性只是影响元素的外观，风格而不影响布局，比如background-color。称为重绘

回流必定引起重绘，反之不成立

触发页面重布局的属性

  • 盒子模型相关属性(width/height,padding,margin,display,border)
  • 定位属性及浮动(top/bottom/left/right,position,float,clear)
  • 改变节点内部文字结构(text-align/weight/family,overflow,line-height,vertical-align,white-space)

只触发重绘的属性

  • color,border-style,border-radius,visibility,text-decoration,background,background-image,background-position,background-repeat,background-size,outline-color,outline,outline-style,outline-width,box-shadow

优化

  • 避免使用触发重绘、回流的css属性
  • 将频繁重绘回流的DOM元素单独作为一个独立图层，那么这个DOM元素的重绘和回流的影响只会在这个图层中

chrome 创建图层的条件

  • 3D或透视变换css属性(perspective transform)
  • 使用加速视屏解码的<video>节点
  • 拥有3D(webGL)上下文或加速2D上下文的<canvas>节点
  • 混合插件(如flash)
  • 对自己的opacity做css动画或使用一个动画webkit变换的元素
  • 拥有加速css过滤器的元素
  • 元素有一个包含复合层的后代元素(一个元素拥有一个子元素，该子元素在自己的层里)
  • 元素有一个z-index较低且包含一个复合层的兄弟元素(元素在复合层上渲染)

总结:

layout:回流， paint:重绘

  • translate替代top(前者少了一个layout过程，可以在chrome的perfomance看到详细过程)
  • opacity替代visibility(前者没layout有paint，后者没layout没paint)
  • 不要一条条修改css，而是放在一个类名里，然后添加这个类名
  • 可以通过display:none;对样式大挑战，然后再display:block;
  • 不要在循环中获取dom位置，可以先把dom位置存下来使用
  • 尽量不使用table布局

  • BFC的原理

浏览器内核及对内核的理解

发表于 2017-08-27 | 分类于 浏览器
浏览器内核

Trident内核：IE
Gecko内核： Firefox
Webkit内核： Safari,Google
Presto内核： Opera，13年之后，Opera宣布加入谷歌阵营，弃用了
Blink内核(基于webkit)： Google和Opera开发的浏览器排版引擎，谷歌还开发了自己的JS引擎，V8，使JS运行速度极大地提高了

对浏览器内核的理解
  • 主要分成两部分：渲染引擎(layout engineer或Rendering Engine)和JS引擎
    • 渲染引擎: 获取页面内容并呈现出来
    • JS引擎： 解析和执行javascript来实现网页的动态效果

Node原理及应用场景

发表于 2017-08-23 | 分类于 Node.js
  • Apache请求数据库
    • res = db.query(‘select * from table’)
    • res.output()
    • 第一行代码会被阻塞
  • node请求数据库
    • db.query(‘select * from table’, function(res) {res.output()})
  • 以上node利用异步回调，就不用等待db.query的结果，而是直接执行接下去的代码
  • 而异步回调会通过事件循环，当有结果了再去调用回调
  • NodeJS处理并发的能力强，但处理计算和逻辑的能力反而很弱，因此，如果我们把复杂的逻辑运算都搬到前端（客户端）完成，而NodeJS只需要提供所有I/O、网络通信等比较耗时的操作
  • 优点:
    • 因为Node是基于事件驱动和无阻塞的，所以非常适合处理并发请求
    • Node代理服务器交互的客户端代码是由javascript语言编写的，
      因此客户端和服务器端都用同一种语言编写，这是非常美妙的事情。
  • 缺点:
    • Node是一个相对新的开源项目，所以不太稳定，缺少足够多的第三方库支持
